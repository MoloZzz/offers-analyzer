# Як працює система вигідних авто — простими словами

> Пояснення для бізнесу, без коду. Дві частини: **(1)** як ми збираємо дані, **(2)** як вирішуємо, що
> оголошення вигідне. Технічні деталі — у [[overview]] та [[profitability-definition]].

Два залізні правила, від яких відштовхуємось: **не загубити вигідне** (щоб через збій не пропустити
хорошу пропозицію) і **не завалити оператора сміттям** (щоб алерти були рідкі й по ділу).

---

## Частина 1. Як ми працюємо (звідки й скільки даних беремо)

**Джерело — офіційне API AUTO.RIA** (developers.ria.com). Не парсимо сайт, працюємо тільки через
офіційний інтерфейс. Причина: легально, стабільно, не порушуємо правила.

**Головне обмеження — безкоштовний тариф ≈ 30 запитів на годину.** Це визначає майже всі рішення: ми
мусимо бути ощадливими й не «палити» ліміт даремно.

**Як часто дивимось:** кожні ~10 хвилин запускається цикл перевірки по всіх наших нішах (профілях).

**Що робимо за один цикл (і скільки це запитів):**
1. **Пошук** — 1 запит на нішу. Повертає лише **список ID оголошень** (без деталей). Дешево, тому робимо
   першим.
2. **Деталі** — по 1 запиту на **кожне нове** оголошення (яке ще не бачили). Повертає повну картку: марка,
   модель, рік, пробіг, ціна, тип продавця (приватник/дилер), VIN і чи є VIN-перевірка, опис, і «прапорці»
   ризику (після ДТП, нерозмитнена, під кредитом тощо).
3. **Середня ціна ринку** — запит на «когорту» (група схожих авто) для розрахунку справедливої ціни.
   **Кешуємо на добу**, тож той самий сегмент не перезапитуємо.
4. **Перевірка знижок** — кілька вже відомих оголошень щоцикл перезапитуємо, щоб зловити зниження ціни.

**Щоб не втратити й не переплатити лімітом:**
- Обробляємо ніші **по колу (round-robin)** і не більше ~15 нових оголошень з однієї ніші за цикл — щоб
  одна широка ніша не з'їла весь ліміт, а інші не лишились без уваги.
- Коли ліміт вичерпано (або API відповів «забагато запитів») — цикл **акуратно зупиняється й
  продовжить наступного разу**, нічого не ламаючи.
- Лічильник ліміту **зберігається в базі**, тож переживає перезапуск і точно відповідає реальному
  використанню.
- Оголошення, яке через збій ліміту не встигли оцінити, **не губиться** — наступного циклу воно перше в
  черзі на оцінку.

**Що ми записуємо в базу (і навіщо):**
- **Оголошення** + **історію цін** — щоб бачити зміни й ловити зниження.
- **Опис** оголошення — щоб перечитувати сигнали стану без повторного запиту до API.
- **Середні ціни** (поточну + історію) — основа розрахунку вигоди й аналізу трендів.
- **Вигідні пропозиції** та **оцінку кожного** оголошення (бал) — щоб показувати найкраще й самонавчатись.
- **Результати** (👍/👎 від оператора, «ціна впала») — «правда життя», на якій система тюнить себе.
- Курс валют беремо щодня з **НБУ** (для перерахунку UAH↔USD).

**Чому саме так (коротко):** ліміт 30/год → беремо спершу дешевий список, деталі лише для нових, середні
ціни кешуємо; ощадливо й нічого не втрачаємо. Історію зберігаємо → можемо ловити знижки й вчитись.
Ліміт у базі → надійно після перезапусків.

## Частина 2. Як ми оцінюємо «вигідність»

Ідея проста: **порівнюємо ціну оголошення з реальною ринковою ціною такого ж авто** і, якщо суттєво
дешевше й без червоних прапорців, — це кандидат у вигідні.

**Крок 1. Ринкова (справедлива) ціна.** Беремо середню ціну AUTO.RIA по «когорті» — авто тієї ж марки,
моделі, року (±1) і, коли даних достатньо, того ж діапазону пробігу. Використовуємо **стійке середнє**
(відкидає викиди — одне абсурдно дороге/дешеве авто не спотворює середню). Місто навмисно не звужуємо —
інакше вибірка стає надто малою.

**Крок 2. Поправка на пробіг.** Якщо порівнювали не з тим самим діапазоном пробігу — коригуємо ринкову
ціну: більший пробіг для віку → трохи знижуємо справедливу ціну (авто варте менше), і навпаки.

**Крок 3. Знижка.** `(ринкова − ціна оголошення) / ринкова`. Наприклад, ринкова $16 000, просять
$13 000 → знижка ≈ 19%.

**Крок 4. Підсумковий бал від −1 до +1.** Це не просто знижка, а три множники разом:
- **знижка** (що дешевше за ринок — то вищий «сирий» бал; ~30% і більше = максимум);
- **впевненість** (скільки схожих авто стояло за середньою: мало даних → бал тягнеться до 0, бо ми не
  впевнені);
- **штраф за ризики** (кожен «м'який» прапорець трохи знижує бал).
Якщо спрацював **дискваліфікуючий** ризик (напр. після ДТП, під кредитом, конфіскат, підозріло велика
знижка >45%) — бал примусово ≤ 0 (це не вигода, а пастка).

**Червоні прапорці** беремо з двох джерел: офіційні позначки AUTO.RIA (ДТП, нерозмитнена, під кредитом,
за кордоном…) і **розбір опису** (укр+рос: «після ДТП», «потребує ремонту», «не на ходу» тощо). Плюс
захист від скрученого одометра: «завелика знижка без VIN-перевірки» і «підозріло малий пробіг для віку».

**Коли це «вигідне» (алерт):** бал ≥ порогу профілю (типово **0.63 ≈ 19% нижче ринку** при повній
впевненості) **і** достатньо даних для довіри **і** немає дискваліфікуючого ризику. Інакше — не алертимо,
але оголошення лишається у списку «найкращих оцінених» (`/best`), тож нічого не зникає.

**Як тримаємо баланс «не загубити / не заспамити»:**
- **Не загубити:** оцінюємо кожне оголошення (не лише ті, що пройшли поріг); недооцінені через ліміт
  переоцінюємо; є `/best` (найкраще навіть нижче порогу) і `/report` (що майже дотягнуло).
- **Не заспамити:** поріг відсікає слабке; ризики гасять пастки; сповіщення ідемпотентні (одне авто —
  один алерт).

**Самонавчання (щоб з часом було точніше):** оператор реагує 👍/👎 на алерти; система рахує реальну
точність і **сама підказує/підкручує поріг** (у межах ±0.1 за раз, з можливістю відкату), а також вчиться,
наскільки штрафувати ризики. За замовчуванням — режим «пропозиція» (нічого не міняє без підтвердження).

## Пояснюваність (чому саме такий бал)

Будь-яку оцінку можна розкласти: команда `/why <посилання>` показує, з якої когорти й на скількох авто
рахували ринкову ціну, поправку на пробіг, розклад балу (`знижка × впевненість × штраф`) і які ризики
спрацювали. Обмеження й що варто додати — див. [[explainability-gaps]].

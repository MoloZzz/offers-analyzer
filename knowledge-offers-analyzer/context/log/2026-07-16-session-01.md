# 2026-07-16 — business-value push: reach non-zero opportunities

## Why

Real runs found **0 вигідних**. Root cause analysed in [[why-no-opportunities]]: the cohort sent to
`/average_price` (make+model+city+exact-year+mileage) collapses the sample to ~1, so the confidence
gate (`sampleSize ≥ 10`) rejects everything. The gate is fine; the cohort was starving it.

## What we implemented (B18 + B19)

1. **Widen the cohort** — new `src/modules/valuation/cohort.ts`: `cohortCandidates` (make+model+year±1,
   then make+model only — city & mileage dropped) + `resolveBenchmark` that walks them until
   `sampleSize ≥ 10`, skipping thin cohorts ("Not Enough Data") instead of dropping the listing.
   Wired into `poll.service.ts` (`evaluateAndNotify`) and `query.service.ts` (`assessById`).
2. **Lower the alert bar** — `defaultMinDealScore` **0.3 → 0.15** (`configuration.ts`, `.env.example`,
   `config/search-profiles.example.json`).
3. **Surface candidates** — `ListingsService.topByScore` + `QueryService.topCandidates` + new bot
   command **`/best`** (lists best-scoring evaluated listings even below the threshold). `/help` updated.
4. **Newest by market** — `submittedWithin` query knob → AUTO.RIA **`top`** submission-period filter
   (plumbed through `SourceSearchQuery`, `ProfileFilters`, `AutoRiaSource.search`, `toQuery`). Example
   profile with **empty `makeModelPairs`** + region + `priceTo` + `submittedWithin: 2` (today), shipped
   **disabled** (budget-heavier — operator opts in). `filters` is jsonb, so no migration.

## Data-backed correction (supersedes B3 assumption)

Checked the AUTO.RIA REST API docs (`AUTO_RIA_API` README): `order_by` only exposes `0` default,
`1` cheap→expensive, `2` expensive→cheap — **no "newest" value**. Freshness is the separate **`top`**
(period) parameter. B3's "newest-first needs order_by verified" is resolved: it's `top`, not `order_by`.

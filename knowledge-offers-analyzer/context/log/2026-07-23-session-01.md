# Session log — 2026-07-23 (01)

## Goal
Resolve backlog item FIX-003.1 (P0, unchecked): verify whether spec 003 Phase 1's
`factorBounds`/`upliftCap` are actually live in prod, since liquidity/repair-risk silently no-op
if they aren't. Pure verification + vault sync — no code changes.

## What was checked
Queried the live Postgres DB (`offers-ria`, table `parameter_sets`) for the active `ParameterSet`
row.

## Result
- Active ParameterSet is **`version 1`**, seeded **2026-07-17** — this predates spec 003 Phase 1,
  which shipped **2026-07-18/19**.
- `factorBounds: null`, `upliftCap: null`.
- `liquidityFactor` (`src/modules/valuation/factors/liquidity.ts`) and `repairRiskFactor`
  (`src/modules/valuation/factors/repair-risk.ts`) both gate on `if (!bounds || !table) return
  null`. With null bounds, both return `null` unconditionally → **`score === priceCore`**,
  identical to pre-spec-003 behavior. Confirmed: liquidity and repair-risk have **zero effect on
  scoring in prod right now**.
- Confirmed **no code path would ever activate them**:
  - `PHASE1_FACTOR_BOUNDS` (`valuation.service.ts:87-90`) is referenced only in test fixtures —
    nothing wires it into a real ParameterSet.
  - `ParametersService.onApplicationBootstrap` seeds a ParameterSet once at startup and never
    re-seeds or updates an existing active row.
  - `CalibrationService.proposeWeights` spreads `{ ...active, ... }` when producing a calibration
    proposal, so the absent `factorBounds` on the active row carries forward untouched into any
    proposed version too.
  - There is no controller, CLI command, or admin endpoint to activate a ParameterSet manually.

This matches exactly what FIX-003.1 anticipated when it was written 2026-07-22 ("Code review
found `factorBounds: {}` in the seed — if so, `score === priceCore`...") — now confirmed against
the live DB rather than inferred from the seed code.

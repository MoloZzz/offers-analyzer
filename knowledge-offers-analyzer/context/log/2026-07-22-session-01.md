# Session log — 2026-07-22

## Goal
Implement a global error handler — the app had no cross-cutting error-handling story: most
Telegram bot commands and 3 of 4 cron jobs had zero `try/catch`, and there was no process-level
safety net.

## Done
1. **Audited the gap** before writing code:
   - `telegram-bot.update.ts`: only `/check` and `/why` had `try/catch`; every other command
     (`/profiles`, `/top`, `/best`, `/report`, `/calibrate`, `/params`, `/revert`, `/weights`,
     `/weights_apply`, `/outcome`, `/blacklist*`, outcome-button action) had none.
   - Cron jobs: only `poll.service.ts` caught its own failures; `calibration-scheduler`,
     `report-scheduler`, `health-monitor` had none.
   - No `uncaughtException`/`unhandledRejection` handlers in `main.ts`.
   - Confirmed via `nestjs-telegraf` internals that bot updates run through Nest's standard
     exception-filter pipeline (`TelegrafExecutionContext`, `host.getType() === 'telegraf'`), so
     one global `@Catch()` filter registered as `APP_FILTER` covers every command/action handler
     without touching each one — no HTTP controllers exist in this app, so no HTTP branch needed.
2. **Implemented** (ADR-0008):
   - `src/common/filters/all-exceptions.filter.ts` — global `AllExceptionsFilter`, logs
     structured via `PinoLogger`, best-effort replies in Ukrainian for `telegraf`-context errors.
   - Registered via `APP_FILTER` in `app.module.ts`.
   - Added `try/catch` to `calibration-scheduler.service.ts`, `report-scheduler.service.ts`,
     `health-monitor.service.ts` — mirrors the existing `poll.service.ts` pattern (catch, log via
     the service's own `PinoLogger`, don't rethrow).
   - `main.ts`: `registerProcessErrorHandlers()` — `uncaughtException` / `unhandledRejection` log
     fatal via the `nestjs-pino` `Logger` instance and `process.exit(1)`.
3. **Tests**: `test/unit/all-exceptions.filter.spec.ts` (4 cases — telegraf reply, reply-failure
   swallowed, non-telegraf context skipped, non-Error throw wrapped). Full suite: 24 suites / 120
   tests pass, `tsc --noEmit` clean, `eslint` clean on all touched files.
4. **Vault updates**: `decisions/0008-global-error-handling.md` (new ADR) + index in
   `decisions/README.md`; `architecture/overview.md` (new "Error handling" line under Stack);
   `conventions/coding-standards.md` (linked ADR-0008 under the "errors are values" principle);
   `operations/environment-setup.md` (new requirement: run under a process supervisor —
   `process.exit(1)` on fatal errors is a no-op safety net without one).

## Decisions / Notes
- Treated this as an infra/technical decision (ADR + direct implementation), not a Spec-Kit
  feature — same precedent as ADR-0007 (structured logging): cross-cutting reliability
  plumbing, no user-facing stories/spec needed.
- `process.exit(1)` on uncaught exceptions is intentional (don't run on possibly-corrupted
  state) but is a new **operational requirement**: the deployment needs a restart-on-crash
  supervisor (systemd/pm2/docker), which isn't yet documented as existing anywhere in this repo
  — `docker-compose.yml` only runs Postgres, the app itself has no restart policy configured.
  Flagged in `environment-setup.md`; not verified whether the user's actual deployment already
  has this.

## Design decisions

- **D1 — dedicated table, not `outcomes`.** A deal is *stateful* with structured money; `outcomes`
  is a label stream. Overloading it would force US7.3 to re-fold label sequences and would risk
  moving spec-002's live threshold prematurely. Separate table ⇒ `closedDeals()` is one indexed
  query and spec-002 is provably unaffected (FR-710 / SC-704).
- **D2 — no telegraf state machine** (the codebase has none). Numbers via one `/deal` line;
  the only two-step interaction (decline → reason) is a stateless inline keyboard carrying the
  opportunity id in `dl:r:<reason>:<opId>` (51 bytes < 64).
- **Stage monotonicity** lives in one pure `deriveStage`. A 🛒 tap carries an explicit
  `intent:'bought'` (no price yet) so it overrides an earlier decline; a `declined` intent never
  overrides a bought/sold deal — unit-tested for every transition. (Caught by the fake-repo test:
  a bare empty-patch `markBought` didn't override a decline until `intent` was added.)
